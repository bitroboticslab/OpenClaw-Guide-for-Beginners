# 发布流程规范

本文档定义本项目的版本管理与发布流程，保证内容更新的一致性和可追溯性。

## 🎯 版本命名规则
采用语义化版本号 `主版本号.次版本号.修订号`，版本号规则：
- **主版本号**：不兼容的重大变更（比如整体结构重构、支持 OpenClaw 新的大版本带来的不兼容变更）
- **次版本号**：向下兼容的功能新增（比如新增平台教程、新增章节内容）
- **修订号**：向下兼容的问题修正（比如修正错别字、更新失效链接、同步小版本功能变更）

### 版本示例：
- `v2.0.0`：整体重构，适配 OpenClaw 2026.0 大版本
- `v1.3.0`：新增 Docker 部署教程、飞书一键对接功能
- `v1.2.1`：修正 Windows 安装脚本链接错误

## 🔖 版本发布周期
- **常规更新**：每月1号发布月度版本，同步当月所有内容更新
- **紧急更新**：遇到内容失效、安全问题、官方重大功能更新时，随时发布修订版本
- **重大更新**：OpenClaw 发布不兼容的大版本更新后，1周内发布适配的主版本更新

## ✅ 发布前检查清单
发布新版本前必须完成以下检查：
1. [ ] 所有内容已对齐 OpenClaw 最新官方版本
2. [ ] 所有内部链接可正常访问
3. [ ] 所有外部链接可正常访问
4. [ ] 所有命令可正确执行
5. [ ] CHANGELOG.md 已更新，包含当前版本的所有变更
6. [ ] 版本号已更新到所有涉及的文档中

## 🚀 发布流程
### 1. 准备发布分支
```bash
# 从main分支创建发布分支
git checkout main
git pull origin main
git checkout -b release/v<版本号>
```

### 2. 更新版本信息
- 更新 `CHANGELOG.md`，添加新版本的变更记录
- 更新文档中所有提到版本号的地方，对齐新版本

### 3. 执行发布检查
```bash
# 执行CI检查脚本
bash scripts/ci-check.sh
```
确认所有检查项通过。

### 4. 提交版本变更
```bash
git add .
git commit -m "chore: release v<版本号>"
git push origin release/v<版本号>
```

### 5. 创建PR审核
创建PR从 `release/v<版本号>` 到 `main` 分支，审核通过后合并。

### 6. 打标签发布
```bash
git checkout main
git pull origin main
git tag -a v<版本号> -m "Release v<版本号>"
git push origin v<版本号>
```

### 7. 发布Release说明
在GitHub上创建Release，内容复制CHANGELOG.md中对应版本的变更记录。

## 🔄 重大更新处理流程
当OpenClaw官方发布重大版本更新（主版本号变更、不兼容的功能变更）时：
1. 24小时内创建紧急Issue，跟踪适配进度
2. 评估变更影响范围，制定适配计划
3. 创建独立的适配分支进行内容更新
4. 完成适配后发布新的主版本，同步所有内容
5. 在README中添加版本兼容性说明，告知用户对应关系

## ⚠️ 回滚机制
如果发布后发现严重问题：
1. 立即回滚main分支到上一个稳定版本
2. 删除有问题的版本标签
3. 创建Issue跟踪问题，修复后重新发布
4. 在CHANGELOG中记录回滚说明

## 📊 版本维护周期
- 最新版本：持续维护，同步所有官方更新
- 历史版本：保留12个月，超过12个月的版本不再更新，提示用户升级到最新版本
- 重大版本：保留至少24个月，提供迁移指南
